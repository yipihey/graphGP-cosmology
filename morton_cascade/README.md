# morton_cascade

Hierarchical multi-statistic spatial analysis for point catalogs, with native
two-catalog (data + randoms) survey-geometry handling.

From a single sparse Morton-ordered dyadic cell hierarchy, the library produces
a wide family of spatial statistics — counts-in-cells distributions, density
moments, two-point correlations, three-point cumulants, wavelet/scattering
coefficients, and survey-footprint diagnostics — at every dyadic scale in one
pass. No mesh required. Empty regions cost nothing.

## What you get from one cascade construction

- **Counts-in-cells (CIC):** `P_N(V)` cell-count PMFs, `P(δ; R)` density-contrast
  distributions, void probability functions
- **Moments:** variance `σ²(R)`, Schur residuals `ΔV_l`, skewness `<δ³>(R)`,
  kurtosis, reduced cumulants `S_3(R)`
- **Two-point:** pair counts `DD/RR/DR(R)`, Landy-Szalay `ξ(R)`, power-spectrum-
  equivalent observables
- **Three-point and higher:** cube-window-averaged bispectrum equivalent
  `<δ³>(R)`, higher cumulants
- **Wavelet/scattering:** axis-aligned Haar wavelet variance per direction,
  anisotropy quadrupole `Q_2`; first-order scattering structure with
  second-order coefficients planned
- **Footprint diagnostics:** outside-footprint mass per scale, active-cell
  counts, effective-volume tracking
- **Cross-statistics:** all of the above between two tracer catalogs

## Why a cascade rather than a mesh?

Conventional FFT-based clustering codes commit to a grid resolution at the
start (memory ∝ `N_cells^D`, FFT cost ∝ `N_cells^D log N_cells`), produce one
set of statistics at one resolution, and need separate machinery for survey
footprints. The cascade is structurally different:

1. **No mesh.** Bit-vector encoding adapts to the point cloud; empty regions
   cost nothing. No grid-resolution decisions, no discretization artifacts.
2. **All scales in one pass.** Dyadic structure produces every scale from
   box-size down to the finest cell in one traversal, with internal consistency.
3. **Footprint-native.** Two-catalog (data + randoms) construction is built
   in. The randoms encode the survey selection function; cell counts of randoms
   weight every estimator by effective volume.

Most cascade statistics scale as `O(N log N)`, with very small prefactors. In
benchmarks against the `HIPSTER` C++ pair-count power-spectrum estimator
(Philcox & Eisenstein 2019): ~30× speedup on `P(k)` monopole, ~150× on the
bispectrum equivalent — and the cascade additionally produces many other
statistics simultaneously. See `examples/hipster_real_benchmark.py`.

## Two ways to use it

This crate ships **both a Rust library and a command-line tool**:

- **Library** (`use morton_cascade::...`) for direct calls from Rust code, MCMC
  inner loops, and other Rust pipelines.
- **CLI** (`morton-cascade <subcommand>`) for shell scripts, pipelines, and
  ad-hoc analysis with point files generated by Python, Julia, C, etc.

## CLI quick start

Install:

```
cargo install --path .
```

Generate a point file from numpy and run the cascade:

```python
import numpy as np
pts = np.random.uniform(0, 256, (50_000, 2)).astype('<f8')
pts.tofile('pts.bin')                         # binary little-endian f64, [N, 2]
```

```bash
morton-cascade cascade -i pts.bin -L 256.0 -o ./out
# → ./out/level_stats.csv
```

Subcommands:

| command       | output                            | notes                                                            |
|---------------|-----------------------------------|------------------------------------------------------------------|
| `cascade`     | `level_stats.csv`                 | mean, var, dvar, sigma^2 per level                               |
| `pmf`         | `level_stats.csv`, `pmfs.csv`     | + per-level cell-count histograms (2D)                           |
| `pmf-windows` | `pmf_windows.csv`                 | `P_N(V)` at log-spaced cube-window volumes                       |
| `tpcf`        | `tpcf.csv`                        | + xi(r) at axis-aligned log-spaced lags                          |
| `pairs`       | `pair_counts.csv`                 | pair counts at all dyadic scales (any D)                         |
| `xi`          | `xi_landy_szalay.csv`             | Landy-Szalay ξ(R) from data + randoms                            |
| `field-stats` | `field_moments.csv`, `field_pdf.csv` | moments and PDF of δ(c) = W_d/(α·W_r) − 1, with footprint cutoff |
| `anisotropy`  | `field_anisotropy.csv`            | cell-wavelet axis-aligned anisotropy moments (3D, first-order Haar scattering) |

reference.

### Multi-run aggregation: shift-bootstrap error bars + non-dyadic scales

The cascade gives statistics at strictly dyadic sides $R_l = 2^{L_{max} - l}$.
The `multi-run` subcommand fills in non-dyadic scales and provides cheap
shift-bootstrap error bars by re-running the cascade across many
shifted-and-rescaled views of the box, then aggregating per-statistic by
physical cell side.

For a periodic cosmological box with shift-bootstrap error bars and 5
points-per-decade scale coverage:

```bash
morton-cascade multi-run -i sim.bin -d 3 -L 1000 -o ./out \
    --statistic anisotropy --boundary periodic \
    --n-shifts 8 --shift-magnitude 0.25 \
    --resize-points-per-decade 5 --resize-min-scale 0.5
```

The cartesian product (8 shifts × ~6 log-spaced rescalings ≈ 48 cascade
runs) is built automatically. Each run is O(N log N), so the whole sweep
is still vastly cheaper than dense O(M^D) sliding-window analyses.

For survey-style isolated boundaries, replace `--boundary periodic` with
`--boundary isolated --randoms randoms.bin`. Resizing then becomes
sub-cube clipping (points outside the rescaled box are dropped); the
`footprint_coverage` column in `multi_run_diagnostics.csv` shows what
fraction of each rescaled sub-cube actually overlaps the survey, so you
can spot runs that walked off the edge.

Each per-bin output row carries an across-run variance for the headline
quantity (`mean_delta_arv` for field-stats, `quadrupole_los_arv` for
anisotropy, `mean_arv` for CIC PMF) — the cheap shift-bootstrap
uncertainty estimate. See `examples/multi_run_demo.rs` for a working
walkthrough.

### ξ(r) with shell deconvolution: the continuous-function fit

For two-point analyses, `--statistic xi` produces shift-pooled-per-resize
pair counts (DD/RR/DR) plus a naive Landy-Szalay ξ per cascade shell.
**Shifts pool**: same shell volume, sum the pair counts (gives shift-bootstrap
variance per shell). **Resize groups stay separate**: different physical
shell volumes can't naively average without losing window information.

To combine resize groups into a smooth shell-deconvolved continuous ξ(r),
add `--xi-fit-basis linear-bsplines`. This fits the underlying ξ(r) by
weighted least squares onto a linear B-spline basis in log r, treating
each cascade shell as a top-hat-windowed measurement of ξ — a Storey-Fisher
& Hogg (2021, ApJ 909, 220) style estimator adapted for the cascade case:

```bash
morton-cascade multi-run -i sim.bin --randoms randoms.bin -d 3 -L 1000 \
    -o ./out --statistic xi --boundary isolated \
    --n-shifts 8 --shift-magnitude 0.25 \
    --resize-points-per-decade 5 --resize-min-scale 0.3 \
    --xi-fit-basis linear-bsplines --xi-fit-knots 16 \
    --xi-fit-window empirical-rr \
    --xi-fit-weighting shift-bootstrap-poisson
```

Outputs:
- `multi_run_xi_raw.csv` — per (resize_group, shell) raw pair counts
- `multi_run_xi_fit_coefs.csv` — fitted basis coefficients with covariance
- `multi_run_xi_evaluated.csv` — fitted ξ(r) on a log-spaced grid

For a Gaussian-process counterpart (RBF kernel, Vecchia-sparse prior via
the [Stanford-ISM `graphgp`](https://github.com/stanford-ism/graphgp)
library by Dodge & Frank), see `examples/xi_graphgp_fit.py`. It takes
the same `multi_run_xi_raw.csv` as input and produces a comparable
fitted ξ(r) using JAX-based GP regression — a useful cross-check
against the in-Rust B-spline fit.

### Differentiable cascade: per-particle gradients of every statistic

Every cascade observable supports a closed-form gradient with respect
to per-particle data weights. For each particle $i$ and each output
bin (level / pattern / shell), `∂(statistic)/∂w_i^d` is computable in
the same order of cost as one forward pass.

What's covered:

| Observable | Per-level grad | Aggregate-scalar grad | Math |
|------------|----------------|------------------------|------|
| `var_delta` (variance of δ) | `gradient_var_delta_all_levels` | `gradient_var_delta_aggregate` | §1 |
| `m3_delta`, `m4_delta` | `gradient_m{3,4}_delta_all_levels` | `gradient_m{3,4}_delta_aggregate` | §1 |
| `s3_delta` (reduced skewness) | `gradient_s3_delta_all_levels` | `gradient_s3_delta_aggregate` | §1 |
| `mean_w_squared_by_pattern` (Haar variances) | `gradient_anisotropy_all_levels` | `gradient_anisotropy_aggregate` | §2 |
| `DD`, `DR` per shell | `gradient_xi_data_all_shells` (in `XiGradient.dd_grads`/`.dr_grads`) | — | §3 |
| `xi_ls` (Landy-Szalay ξ) | `gradient_xi_data_all_shells` (in `.xi_grads`) | `gradient_xi_data_aggregate` | §3 |

Section references are to `docs/differentiable_cascade.md`, which has
the full algebra and verification strategy.

The aggregate-scalar variant takes per-bin "loss weights" and returns
one $N_d$-vector $\partial L/\partial w_i^d$ where $L = \sum_\text{bin} \beta_\text{bin} \cdot$ statistic.
This is the optimization-friendly path: $O(N_d)$ memory regardless of
how many bins are active.

```rust
use morton_cascade::coord_range::{CoordRange, TrimmedPoints};
use morton_cascade::hier_bitvec_pair::{BitVecCascadePair, FieldStatsConfig};

let pts_d: Vec<[u64; 3]> = /* data catalog */;
let pts_r: Vec<[u64; 3]> = /* random catalog */;
let weights_d: Option<Vec<f64>> = None; // or Some(per-particle weights)

let range = CoordRange::analyze_pair(&pts_d, &pts_r);
let td = TrimmedPoints::from_points_with_range(pts_d, range.clone());
let tr = TrimmedPoints::from_points_with_range(pts_r, range);
let pair = BitVecCascadePair::<3>::build_full(td, tr, weights_d, None, None, 64);

let cfg = FieldStatsConfig::default();
let stats = pair.analyze_field_stats(&cfg);

// Optimize a scalar loss combining variance at multiple cascade levels.
let betas: Vec<f64> = stats.iter().map(|s| s.cell_side_trimmed.ln()).collect();
let grad: Vec<f64> = pair.gradient_var_delta_aggregate(&cfg, &stats, &betas);
// grad[i] is now ∂L/∂w_i^d, ready for an optimizer.
```

For per-particle attribution (which particles influence which output
bin, by how much), the underlying primitive is `cell_membership::CellMembership` —
a Morton-sorted reverse-lookup index providing $O(\log N)$ "which
particles are in cell $c$ at level $\ell$" queries. See
`examples/per_particle_attribution.rs` for a worked CIC PMF example.

**Random-weight gradients** ($\partial/\partial w_j^r$): variance,
$\mu_3$, $\mu_4$, $S_3$, anisotropy $\langle w_e^2\rangle$, and
$\xi_{LS}$ (with constituent DR, RR pair-count derivatives) are all
implemented. DD's random-weight gradient is identically zero (DD
doesn't depend on randoms). Public methods follow the same naming
convention as data-weight: `gradient_*_random_all_levels` and
`gradient_*_random_aggregate`. See `docs/differentiable_cascade.md` §6.

**Multi-run gradients**: per-original-particle gradient of the
**pooled** observables produced by the multi-run aggregator. Available
in both data-weight and random-weight directions, for all three
observables:

- **Field-stats moments** (`var_delta`, `m3_delta`, `m4_delta`,
  `s3_delta`): `gradient_{var,m3,m4,s3}_delta_data_pooled` and
  `gradient_{var,m3,m4,s3}_delta_random_pooled` — each returns one
  gradient vector per pooled output bin. Linear-composition variant
  (run-averaged variance gradient) available as
  `gradient_var_delta_data_run_average`.
- **Anisotropy**: `gradient_anisotropy_data_pooled` and
  `gradient_anisotropy_random_pooled` return per-pattern, per-axis,
  and LoS-quadrupole gradients per pooled bin. Aggregate-scalar
  variants: `gradient_anisotropy_quadrupole_{data,random}_pooled_aggregate`.
- **ξ (Landy-Szalay)**: `gradient_xi_data_pooled` and
  `gradient_xi_random_pooled` return per-shell, per-particle
  gradients within each resize group. Aggregate-scalar variants:
  `gradient_xi_{data,random}_pooled_aggregate`. Note: differentiates
  the count-based pooled normalization that the existing forward
  `pool_xi_resize_group` uses (consistent with the forward, but does
  NOT inherit weight-based uniform-scaling invariance).

Foundational primitives:
- `gradient_raw_sums_data_all_levels` returns per-particle
  $(\partial S_1, \partial S_2, \partial S_3, \partial S_4)$ for the
  data-weight side
- `gradient_raw_sums_random_all_levels` returns
  $(\partial T, \partial S_1, \partial S_2, \partial S_3, \partial S_4)$
  for the random-weight side (the extra $\partial T$ is needed because
  $T$ depends on random weights)

See `docs/differentiable_cascade.md` §7.

**Forward-mode (JVP) and Hessian-vector products** (`jvp` module):
helpers for forward-mode directional derivatives and HVPs built on
top of the existing reverse-mode gradient infrastructure. The JVP
helpers are thin wrappers (one row-dot-product per output) around
the pooled gradient structs; their value is exposing "directional
derivative of all observables in direction $v$" as a named API.
The `hvp` helper computes Hessian-vector products via central
finite differences over the gradient — two gradient evaluations,
no Hessian materialized — enabling Newton-CG, trust-region, and
Fisher-information workflows on top of the cascade.

**Cell pair-separation kernels** (`cell_kernels` module):
relate $\xi(r)$ to cell-averaged variance $\bar\xi_V$ via
$\bar\xi_V = \int K_V(r) \xi(r) dr$. Provides
`AnalyticBoxKernel1D` (closed-form triangular density),
`MonteCarloKernel::unit_cube::<D>(n_pairs, n_bins, seed)` (generic
MC-tabulated kernel for any D ≤ 32), and the integration routines
`variance_from_xi` and `compute_kernel_betas_for_group`. The
kernel-weighted variance is **differentiable for free**: kernel
weights `betas[s] = K_V(\bar r_s) · Δr_s` plug directly into the
existing `gradient_xi_{data,random}_pooled_aggregate` API, no new
gradient code required. Useful for cross-checking the cascade's
two variance estimators (`var_delta` directly vs kernel-from-ξ),
or for predicting $\sigma^2(V)$ from a theoretical $\xi(r)$.

**Worked optimization examples**:

- `examples/optimize_data_weights.rs` — Adam on the pooled-variance
  gradient with absolute-error loss `L = Σ_b (var_b − target_b)²`.
  Loss decreases ~10000×; large bins converge to <1% but small bins
  stick at 15-30% relative error (the loss correctly weights them
  less in absolute terms).
- `examples/optimize_data_weights_logloss.rs` — same setup with
  log-space loss `L = Σ_b (log var_b − log target_b)²` for equal
  fractional error per bin. All bins converge to ≤0.4% relative
  error. Demonstrates the practical considerations: var-floor for
  gradient-blowup protection, smaller learning rate, larger FD eps.
- `examples/optimize_data_weights_newton_cg.rs` — Newton-CG using
  HVPs for second-order curvature. Converges to ≤0.01% relative
  error in every bin, ~10¹³× loss reduction at machine precision in
  ~25 outer iterations. Demonstrates how truncated-CG handles
  negative curvature (frequent on squared-residual losses far from
  the optimum) and rank-deficient Hessian (the uniform-scaling flat
  direction in log-weight space).

## Library quick start

```rust
use morton_cascade::hier;

// Coordinates are u16 per axis, interpreted as the unit periodic box scaled to [0, 2^16).
let pts: Vec<(u16, u16)> = /* your point cloud */;

// Periodic, with sub-shift refinement (s_subshift = 1 is the usual choice).
let (level_stats, n_distinct) = hier::cascade_hierarchical_bc(&pts, 1, true);

for (l, st) in level_stats.iter().enumerate() {
    let R = 1usize << (morton_cascade::L_MAX - l);
    let sigma2 = (st.var - st.mean) / (st.mean * st.mean);
    println!("R = {R:>3}  <N> = {:>10.2}  sigma^2 = {sigma2:.4}", st.mean);
}
```

For two-catalog (data + randoms) survey work — the main use case — see
`hier_bitvec_pair::BitVecCascadePair` which exposes `analyze_field_stats`,
`analyze_anisotropy`, and pair-count utilities.

## Performance

On a single-threaded modern x86 CPU (rough numbers from the development sandbox):

| dimension | N         | cascade time | crossover vs Corrfunc |
|-----------|-----------|--------------|------------------------|
| 2D        | 100k      | ~30 ms       | Corrfunc faster below ~1M |
| 2D        | 800k      | ~60 ms       | cascade ~30x faster   |
| 2D        | 50M       | ~200 ms      | cascade ~100x faster  |
| 3D        | 200k      | ~3.4 s       | parity                |
| 3D        | 2M        | ~3.4 s       | cascade ~24x faster   |
| 3D        | 100M      | ~5 s (extrap) | cascade ~5000x faster |

The cascade is constant-time in N above its working-set threshold (~M^D × L_max
operations). Pair-counting xi(r) implementations scale as N^1.6 in 3D at fixed
density.

**Comparison fairness**: the cascade numbers above are single-threaded.
Corrfunc internally uses OpenMP, so the comparison understates how
much faster a parallel cascade run would be. The single-cascade build
itself remains single-threaded; parallelism enters only at the
multi-run aggregator level (see below).

**Multi-run parallelism**: `CascadeRunner::per_run` and the six
multi-run analyses/gradients (`analyze_field_stats`,
`analyze_anisotropy`, `analyze_cic_pmf`, `analyze_xi`,
`gradient_var_delta_data_run_average`, and the four pooled-moment
gradients) parallelize across runs via rayon. Near-linear speedup up
to N_runs threads. Outputs are bit-exact reproducible across thread
counts (collect-then-serial-reduce pattern). See `docs/parallelism.md`
for the audit and design rationale, and
`examples/profile_multi_run.rs` for measuring speedup on your hardware.

## Algorithm notes

At every level l:
- Cell side R_l = 2^(L_max - l) in tree-coord units
- Number of cells = (2^l)^D
- Mean count per cell = N / (2^l)^D
- Variance and Schur residual computed from a hierarchical (cell, shift) buffer

For uniform Poisson input, the Schur ratio dvar/<N> equals
`1 - (2^D - 1) / 4^D` exactly: 13/16 in 2D, 57/64 in 3D, 993/1024 in 5D.
This is the theoretical floor; deviations above it are clustering signal.

The cascade uses an "all-shifts on" trick: instead of running the algorithm
separately for each origin shift, all shifts are encoded in a single buffer of
size M^D where M = 2^(L_max + s_subshift). This makes the per-level cost
independent of the number of shifts, with `s_subshift = 1` (twice the finest
resolution) being the usual choice.

## Coordinate convention

Input points are unsigned 16-bit per axis. Map your physical coordinates as
`x_u16 = (x_phys / box_size * 65536.0) as u16`. The periodic domain is the unit
box [0, 1), wrapping at 2^16.

## Testing

```
cargo test --release          # ~45 s, runs all 14 integration tests
cargo test --release -- --nocapture   # show test output
cargo doc --no-deps           # build API docs
```

The test suite covers:
- Conservation: per-level mean exactly equals N / 2^(D·l)
- Schur ratio matches `1 - (2^D-1) / 4^D` at D = 2, 3, 4, 5
- Generic `cascade_nd<D>` is bitwise-identical to hand-written 2D and 3D
- Packed (adaptive-width) cascade is bitwise-identical to dense reference
  in periodic, non-periodic, and hint/no-hint modes
- Uniform Poisson xi(r) is statistically zero at r > 0
- PMF histograms sum exactly to n_total, first moment matches reported mean
